2026-05-09 — Decision: Fork astro-modular to MMoMM-org/astro-modular-mmomm for i18n customizations — Rationale: upstream update mechanism wipes all theme files outside USER_PATHS on every upgrade; a fork keeps i18n patches in version control and survives upgrades. See [ADR-001](../../XDD/adr/ADR-001-fork-astro-modular-for-i18n.md).
2026-05-09 — Decision: Bilingual i18n architecture — symmetric folders (DE at posts/de/<slug>/, EN at posts/en/<slug>/), Astro core i18n with prefixDefaultLocale:false, single collection with explicit `lang` frontmatter, ship DE+EN together, 301 redirects from /blog/ to /posts/ — Rationale: idiomatic Astro i18n, future-locale-friendly, Obsidian vault tidier; URL strategy unchanged (DE at root, EN at /en/). See [ADR-002](../../XDD/adr/ADR-002-i18n-architecture.md). [Revised same day from asymmetric — see ADR Revisions.]
2026-05-09 — Decision: i18n URL builders for `<script>`-block consumers — pre-compute `url: postUrl(post)` server-side and serialize via `define:vars` in the post-data projection; runtime reads `post.url` directly — Rationale: after the symmetric refactor `post.id` is `<lang>/<bare-slug>`, so any client-side `` `/posts/${post.id}` `` produces `/posts/de/foo` (broken). Browser scripts can't import `astro:content`, so the locale logic must live server-side. Applied in T2 (ProjectLayout), T4 (PostLayout). Don't try to recompute `localePrefix + slug` in the browser script.
2026-05-09 — Decision: Slug→lang resolution for remark plugins — lazy filesystem walk of `src/content/posts/**/*.md`, parse frontmatter `lang:` line via regex, cache on `globalThis` via `Symbol.for` to survive dual-instantiation (config-graph load + runtime). See `buildSlugLangMap` / `getSlugLangMap` in `src/utils/internallinks.ts` — Rationale: remark plugins run at content-bundling time, BEFORE any layout populates `globalPostsCache`. Layout-side caching alone can't help. The filesystem walk is one-time, memoized, and survives the dual-load of `internallinks.ts` (config-graph + runtime). Reusable pattern for any remark plugin needing post metadata that's not yet built.
2026-05-09 — Decision: Translation-aware lookup for single-value config (T11 featured post) — cascade: (a) try direct match in current-locale's posts using bare slug, (b) if missing in current locale, find post across all locales and resolve via `findTranslation(post)` to the locale counterpart, (c) fallback to latest post in current locale — Rationale: `siteConfig.homeOptions.featuredPost.slug` is a single string but the locale-aware site needs a per-locale featured post. Translation lookup keeps editorial intent (DE editor pins a featured post; EN homepage shows its translation automatically). Reusable shape for any single-value config that should feel translation-aware.
2026-05-09 — Decision: Path-based ID generators must use `/` (not `-`) for nested folders — `scripts/generate-graph-data.js` previously joined nested folders with `-` producing `category-my-post`, but Astro's `post.id` uses `/` producing `category/my-post`. Latent mismatch surfaced when the symmetric refactor moved posts under `de/` and `en/` — every migrated post was silently dropped from graph data. Fix: change `generateNodeId` to preserve `/` and apply slug cleanup per-segment — Rationale: any tool whose output is consumed against `post.id` must mirror its format exactly. Same principle applies to any future content-pipeline scripts (search index, related-posts, etc.).
2026-05-09 — Decision: Type-safe i18n table via `Record<Locale, StringMap>` (T9) — single source-of-truth `StringMap` type defines the key set, then `const strings: Record<Locale, StringMap>` enforces every locale supplies every key. `t(locale, key)` is a typed lookup; `StringKey = keyof StringMap` is the public surface. See `src/i18n/strings.ts` — Rationale: missing-translation drift becomes a compile error rather than a runtime fallback or silent display bug. No runtime dependency (no i18next), no JSON files, no codegen — a small TS module that the type system polices. Reusable for any future translation table (e.g. content-collection-typed labels). Interpolation helper deferred until needed.
2026-05-11 — Decision: Multi-locale architecture (N-locale-ready design, MVP stays DE/EN) — Six structural decisions: (1) keep `Locale = 'de' \| 'en'` union for MVP, all APIs N-shaped so widening = one line; (2) `siteConfig.locales` + `defaultLocale` become single source of truth, `LOCALES`/`DEFAULT_LOCALE` re-source from there, legacy `language: "de"` removed; (3) `LocalisedString = Record<Locale, string>` required-all with `lt(locale, value)` helper for migration safety; (4) `urlEn` → `urlByLocale: Partial<Record<Locale, string>>`, resolver centralised in `i18n.ts`; (5) routes parameterised as `src/pages/[locale]/...` over non-default locales, killing 7-file duplication under `src/pages/en/`; (6) plugin fork (`MMoMM-org/astro-modular-settings-mmomm`) deferred to Phase 3 and built locale-aware from the start to avoid double-rewrite. ADR-002 Decision 4 revised (parameterised routes), ADR-003 Decision 3 revised (`urlByLocale` replaces `urlEn`), ADR-004 Decision 1 reaffirmed (Vault CMS scales linearly). See [ADR-005](../../XDD/adr/ADR-005-multi-locale-architecture.md) — Rationale: user signalled "kompletter Umbau auf multi language" with future xx/yy/zz locales; doing the work twice (bilingual now, multi-locale later) costs strictly more than once. TypeScript enforces locale parity at compile time on every `LocalisedString` and per-locale formatter table.

Three known cliffs documented for N>2 (bilingual `findTranslation`/`findPageTranslation`, single-link switcher in Header.astro, hreflang fan-out verification). ADR-005 Status flipped Proposed→Accepted with Phase 3 fork-commit cross-links (`c816b24`/`8b73433`/`d6596b4`/`ef6ac68` + vault `39ae100`). `// [CONFIG:NAVIGATION_FOOTER]` marker added immediately above `footer:` in src/config.ts so the plugin's marker-anchored index-scan no longer relies on structural detection — Rationale: ADR-005 is now fully shipped and operationally documented; the runbook makes adding locale #3 a checklist exercise rather than a research expedition; the marker addition future-proofs round-trip against potential reordering of unrelated nav fields. 2026-05-12 — Decision: ADR-005 Phase 3 shipped — plugin re-enable safe — Four-commit fork `MMoMM-org/astro-modular-settings-mmomm` `feat/multi-locale-aware` (c816b24, 8b73433, d6596b4, ef6ac68) implements i18n-aware round-trip for `astro-modular-settings`. Architectural keys, learned the hard way over 4 smoke-test rounds: (a) **Load-time overlay in `loadSettings()`** — parse `config.ts` after `loadData()` and overwrite `settings.navigation.pages`/`footer`, LocalisedString site-info, `locales`, `defaultLocale`, `footer.content` from there; without this, a stale `data.json` (e.g. nav stripped of `i18nKey`) propagates to the serializer and round-trips back into config.ts, defeating any passthrough work. (b) **In-place array rewrite** — find `<field>: [` after the marker and replace ONLY the `[...]` body via depth-tracked closing-bracket scan; the previous "wipe everything between markers" strategy wiped sibling marker-less fields (the `footer: [...]` block had no `[CONFIG:NAVIGATION_FOOTER]` marker and lived between `NAVIGATION_PAGES` and `NAVIGATION_SOCIAL`, so it got nuked). (c) **Comment-aware value parser** — `parseTsArray`/`parseTsObject` skip `// ...` and `/* */` wherever they skip whitespace; otherwise embedded doc-comments between array items get parsed as a stream of `{__raw: identifier}` garbage items. (d) **Marker-anchored index-scan replaces regex** — `\s*\n\s*<field>:` regex fails when documentation comments sit between marker and field; switching to `indexOf(marker)` → search for `\bfield\s*:\s*\[` → `parseTsArray` from `[` is tolerant of intervening content. Smoke-tested by toggling `showSocialIconsInFooter` through the plugin UI: all 6 `i18nKey`, 3 `urlByLocale`, 1 `external:true`, the `navigation.footer` array (2 items with i18nKey/urlByLocale), `locales`, `defaultLocale`, and the LocalisedString title/description/footer.content shapes survived intact. Release-mirror workflow (analog ADR-001 Deferred #1) and Phase 4 runbook deferred. See ADR-005 Phase 3 + fork `feat/multi-locale-aware` branch.
