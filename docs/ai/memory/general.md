## Content authoring: pages start body at H2, never H1
<!-- 2026-05-11 -->
`src/layouts/PageLayout.astro` (and `PostLayout.astro`) renders the frontmatter `title` field as the page's `<h1>` (`PageLayout.astro:152-154`). If the body markdown opens with `# Heading`, the rendered page has two H1s — bad for SEO and the page outline. Always start `pages/**/*.md` and `posts/**/*.md` bodies at H2 (`##`) or lower. When migrating from systems that put the title in the body (Hugo, Wordpress export, plain-text drafts), remove or downgrade the redundant body H1. The same applies to `posts` — keep the visual hierarchy consistent (PostLayout owns the page H1).

## Path-based slug detection must skip the locale folder
<!-- 2026-05-11 -->
ADR-002 puts i18n content under `src/content/<collection>/<locale>/<slug>/` (locale = `de`|`en`). Any code that derives a slug from a file path by doing `pathParts[pathParts.indexOf(collection) + 1]` will pick up the **locale folder** as the slug for i18n content and silently mis-route URLs (`/posts/en/<file>` instead of `/en/posts/<slug>/<file>`). When adding a new remark plugin, content helper, or pipeline step that walks paths, peek at `pathParts[collectionIndex + 1]`: if it's `'de'` or `'en'`, the real slug is at `+ 2` and the URL needs a `/en` prefix for EN (empty for DE). Existing precedents: `remarkFolderImages` in `internallinks.ts`, both branches of `remark-obsidian-embeds.ts`, and the `syncFolderBasedImages` function in `scripts/sync-images.js` (which routes output to `public/<locale-prefix>/<collection>/<slug>/<file>` so URLs and files match).

## Locale-neutral pages: special collection + dedicated route + switcherHref override
<!-- 2026-05-11 -->
Some pages don't fit the bilingual symmetric layout — a single URL shared across locales, content in one language, no translation counterpart. The first example is `/now/` (a "Now Page Movement" page, English-only). Convention for this kind of page (precedent: `src/pages/now.astro`, `src/content/special/now.md`):

1. **Content lives in `src/content/special/`** — the special schema has no `lang` or `translationKey` field, so the bilingual machinery (`findPageTranslation`, hreflang emission) stays out of the way.
2. **Route is a dedicated `.astro` file under `src/pages/`** (not generated by `[...slug].astro`). Add the special-entry id to the filter list in `src/pages/[...slug].astro:31-35` to suppress the "Could not render … conflicts with higher priority route" build warning.
3. **Nav config sets both `url` and `urlEn` to the same value** — without `urlEn`, the Header would prepend `/en/` on EN routes and break the link. Example: `{ title: "Jetzt", i18nKey: "nav.now", url: "/now/", urlEn: "/now/" }`.
4. **Route passes `switcherHref="..."` to `BaseLayout`** — the default `pathForOtherLocale` computation would produce `/en/<slug>/` which doesn't exist. Pick the most reasonable target for the other locale (typically the other locale's homepage, e.g. `switcherHref="/en/"` on `/now/`).

Trade-off accepted: arriving at the page from EN nav lands on a URL with no `/en/` prefix, so the Header re-derives DE locale and renders DE nav labels around the (English) content. Acceptable for deliberately locale-shared pages; not acceptable for content that should switch UI labels per visitor — those need a proper bilingual page pair under `src/content/pages/{de,en}/`.

## Tag-index pages are frontmatter-only — inline `#tags` don't auto-register
<!-- 2026-05-12 -->
`extractTags(posts)` in `src/utils/markdown.ts:369` reads ONLY `post.data.tags` (frontmatter). The `getStaticPaths` in both `src/pages/posts/tag/[...tag].astro` and `src/pages/[locale]/posts/tag/[...tag].astro` is driven by that list. Consequence: a `#FooBar` written inline in a body via `remarkInlineTags` produces a clickable `<a href="/posts/tag/foobar">` link, but no `/posts/tag/foobar/` page exists unless at least one post also has `foobar` in its frontmatter `tags:` array. The `href` is lowercased by the plugin (`src/utils/remark-inline-tags.ts`) so it matches the lowercase-by-convention frontmatter tags on case-sensitive deploy targets (Linux, Netlify). Two options when adding new inline tags: (a) also list them in the relevant posts' frontmatter `tags:` (canonical pattern), or (b) extend `extractTags()` to harvest from body content too (theme-fork edit, generates Tag pages for body-only tags). Tracked as a future "tag-cloud fallback on unknown tag URL" idea in `context.md`.

## Astro schema `.default(...)` makes that field never nullish — `||` fallbacks always pick the default
<!-- 2026-05-12 -->
`content.config.ts` declares e.g. `description: z.string().nullable().optional().default('No description provided')`. Astro applies the default during validation, so `post.data.description` is ALWAYS a string at runtime — even when the source file's frontmatter has no `description:` key. Consequence: a naive `description || excerpt` fallback in a layout/card component will never reach the excerpt branch — it'll always render the placeholder string. Pattern that works: filter explicitly against the default string before the OR. Example in `src/components/PostCard.astro`: `const hasRealDesc = description && description !== 'No description provided'; const text = hasRealDesc ? description : excerpt;`. Same trap applies to any other schema field with a `.default(...)` that consumer code wants to treat as "missing" (e.g. `title.default('Untitled Post')`).

## Cross-cutting components derive locale from URL, not props
<!-- 2026-05-11 -->
Header.astro and Footer.astro both derive the current locale via `Astro.url.pathname` rather than trusting a `lang` prop. The reason is that not every route reliably propagates `lang` through to BaseLayout — historically PageLayout passed nothing (fixed in Track A, but the workaround stays as a safety net), and special-route shells (404, etc.) also rendered without it, which caused the language switcher to compound `/en/en/en/...` on each click. The URL is the canonical source for the current locale on a static site, so any nav-aware component (switcher, locale-prefixed links, locale-keyed T9 lookups) should look at `Astro.url.pathname`. Props remain useful for *content* concerns (which post/page are we rendering) and for layout overrides; they are not the source of truth for "which locale am I in."
