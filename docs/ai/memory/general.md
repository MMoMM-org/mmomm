## Content authoring: pages start body at H2, never H1
<!-- 2026-05-11 -->
`src/layouts/PageLayout.astro` (and `PostLayout.astro`) renders the frontmatter `title` field as the page's `<h1>` (`PageLayout.astro:152-154`). If the body markdown opens with `# Heading`, the rendered page has two H1s — bad for SEO and the page outline. Always start `pages/**/*.md` and `posts/**/*.md` bodies at H2 (`##`) or lower. When migrating from systems that put the title in the body (Hugo, Wordpress export, plain-text drafts), remove or downgrade the redundant body H1. The same applies to `posts` — keep the visual hierarchy consistent (PostLayout owns the page H1).

## Path-based slug detection must skip the locale folder
<!-- 2026-05-11 -->
ADR-002 puts i18n content under `src/content/<collection>/<locale>/<slug>/` (locale = `de`|`en`). Any code that derives a slug from a file path by doing `pathParts[pathParts.indexOf(collection) + 1]` will pick up the **locale folder** as the slug for i18n content and silently mis-route URLs (`/posts/en/<file>` instead of `/en/posts/<slug>/<file>`). When adding a new remark plugin, content helper, or pipeline step that walks paths, peek at `pathParts[collectionIndex + 1]`: if it's `'de'` or `'en'`, the real slug is at `+ 2` and the URL needs a `/en` prefix for EN (empty for DE). Existing precedents: `remarkFolderImages` in `internallinks.ts`, both branches of `remark-obsidian-embeds.ts`, and the `syncFolderBasedImages` function in `scripts/sync-images.js` (which routes output to `public/<locale-prefix>/<collection>/<slug>/<file>` so URLs and files match).

## Special collection is bilingual + fragment-only (no standalone routes)
<!-- 2026-05-12 (replaces 2026-05-11 locale-neutral-pages rule, Phase C) -->
`specialCollection` requires `lang: z.enum(['de','en'])` and uses folder-per-locale layout: `src/content/special/{de,en}/<name>.md`. The collection holds **fragments only** — content embedded into route shells (homepage blurb, posts/projects/docs index intro, 404 body), never standalone URLs. `[...slug].astro` does NOT emit special entries as routes since ADR-005 Phase C (2026-05-12). Lookup pattern: `getEntry('special', `${locale}/${name}`)` (e.g. `getEntry('special', 'de/home')`). The five canonical names are `home`, `404`, `posts`, `projects`, `docs`. **404 is the one known default-locale-only exception**: Astro emits a single static `404.html`, so `src/pages/404.astro` reads `${DEFAULT_LOCALE}/404` — the EN variant exists for future per-locale 404 routes but isn't wired today.

Historical note: there *was* a "locale-neutral page" pattern for `/now/` (`switcherHref` override + `urlByLocale: {de, en} same`). Phase C removed it — `/now/` is now a regular bilingual page pair (`src/content/pages/{de,en}/now.md`, both copies authored in EN, `translationKey: now` links them). If a future page genuinely cannot be translated, prefer authoring two copies of the same English content in the bilingual `pages/` collection over reviving the locale-neutral mechanism — keeps the URL scheme uniform, plays nice with the switcher, and avoids the special-case `switcherHref` plumbing.

## Audit hardcoded URL prefixes (`/posts/`, `/posts/tag/`, ...) in theme code on every new locale
<!-- 2026-05-12 -->
ADR-005 Phase 1 caught the 5 EN-route URL bugs in pages routes, and Phase 2 dedup'd the parallel route tree — but **theme components and utils still contain literal `/posts/tag/...` constructions** (3 found 2026-05-12 in `PostLayout.astro:258`, `Tags.astro:28`, `seo.ts:292`). All emitted bare DE URLs on EN pages until fixed. **Rule**: whenever locale-related work happens, grep `grep -rn '/posts/\|/posts/tag/\|/projects/\|/docs/' src/ --include='*.astro' --include='*.ts'` and verify each hit either (a) uses a locale-aware builder (`postUrl`, `tagUrl`, `pageUrl`, `navUrl` — all in `src/utils/i18n.ts`), (b) is inside a default-locale-only route (DE pages where bare URLs are correct), or (c) is inside a `[locale]/` parametric route that already prepends `${prefix}`. The same audit applies to `<a href="/...">`, `Astro.redirect('/...')`, and SEO/sitemap canonical builders. Single source of truth: prefer the `*Url(locale, ...)` helpers from i18n.ts over inline construction.

## Tag-index pages are frontmatter-only — inline `#tags` don't auto-register
<!-- 2026-05-12 -->
`extractTags(posts)` in `src/utils/markdown.ts:369` reads ONLY `post.data.tags` (frontmatter). The `getStaticPaths` in both `src/pages/posts/tag/[...tag].astro` and `src/pages/[locale]/posts/tag/[...tag].astro` is driven by that list. Consequence: a `#FooBar` written inline in a body via `remarkInlineTags` produces a clickable `<a href="/posts/tag/foobar">` link, but no `/posts/tag/foobar/` page exists unless at least one post also has `foobar` in its frontmatter `tags:` array. The `href` is lowercased by the plugin (`src/utils/remark-inline-tags.ts`) so it matches the lowercase-by-convention frontmatter tags on case-sensitive deploy targets (Linux, Netlify). Two options when adding new inline tags: (a) also list them in the relevant posts' frontmatter `tags:` (canonical pattern), or (b) extend `extractTags()` to harvest from body content too (theme-fork edit, generates Tag pages for body-only tags). Tracked as a future "tag-cloud fallback on unknown tag URL" idea in `context.md`.

## Astro schema `.default(...)` makes that field never nullish — `||` fallbacks always pick the default
<!-- 2026-05-12 -->
`content.config.ts` declares e.g. `description: z.string().nullable().optional().default('No description provided')`. Astro applies the default during validation, so `post.data.description` is ALWAYS a string at runtime — even when the source file's frontmatter has no `description:` key. Consequence: a naive `description || excerpt` fallback in a layout/card component will never reach the excerpt branch — it'll always render the placeholder string. Pattern that works: filter explicitly against the default string before the OR. Example in `src/components/PostCard.astro`: `const hasRealDesc = description && description !== 'No description provided'; const text = hasRealDesc ? description : excerpt;`. Same trap applies to any other schema field with a `.default(...)` that consumer code wants to treat as "missing" (e.g. `title.default('Untitled Post')`).

## Never hand-roll URLs from `entry.id` — IDs include the locale folder
<!-- 2026-05-12 -->
Astro Content Collection IDs for posts/pages include the locale folder prefix (e.g. `de/obsidian-todoist`, `en/the-2-aspects-of-miyo`) because of the symmetric `src/content/<collection>/<locale>/<slug>/` layout from ADR-002. Any dynamic URL construction like `` `/posts/${post.id}` `` or `` `/${page.id}` `` therefore produces broken paths (`/posts/de/...` instead of `/posts/...`; `/posts/en/...` instead of `/en/posts/...`). This bug pattern has shown up twice in production: in `src/pages/api/{posts,pages}.json.ts` (search command palette pointed at 404s, fixed in `2e977ba`) and in the wikilink resolver `src/utils/internallinks.ts` (cross-post wikilinks rendered to `/posts/<lang>/<slug>` 404s, fixed in `4eb89eb`). **Rule**: never compose post/page URLs from `entry.id` directly — always go through `postUrl(post)` / `pageUrl(page)` from `src/utils/i18n.ts`, which strip the locale folder and add the right prefix (`''` for default DE, `/en` for EN). Same applies to any new helper that derives a URL from a content entry. Companion rule for hardcoded URL prefixes is above ("Audit hardcoded URL prefixes ..."), but the `entry.id` template-string form slips past that grep because the path looks structural — code review must specifically flag any `${entry.id}` interpolation inside a URL.

## Cross-post wikilinks use vault-relative path form
<!-- 2026-05-12 -->
Cross-post wikilinks must be authored as `[[posts/<lang>/<slug>/index|<Display Text>]]` (vault-relative path form, with `index` because every post is a folder with `index.md`), not the bare-slug form `[[<slug>|<Display>]]`. Reason: every post folder ships an `index.md`, so DE and EN translations of the same post share the same Obsidian filename (e.g. both `posts/de/miyo-ace/index.md` and `posts/en/miyo-ace/index.md` resolve to "miyo-ace" by name). The bare-slug form is ambiguous — Obsidian picks whichever it indexed first, and the build-time wikilink resolver in `src/utils/internallinks.ts` fell back to a locale-blind slug-lang map that mis-routed cross-locale slugs (see troubleshooting.md "slug-lang-map locale-blind on duplicate slugs"). The vault-relative path form is unambiguous: it encodes both the target locale and the slug. Pattern is what the Obsidian Astro Composer plugin generates when you wikilink via path completion (`[[ ` + start typing path), and what the existing fixed wikilinks in `obsidian-buttons` already use. The resolver (`resolveLocaleAwarePostUrl()` in `internallinks.ts:97`) trusts the explicit locale prefix in the path and emits `/posts/<slug>` for `de/...` and `/en/posts/<slug>` for `en/...` — so the format is also resilient if the slug-lang map ever has duplicate-key collisions.

## Cross-cutting components derive locale from URL, not props
<!-- 2026-05-11 -->
Header.astro and Footer.astro both derive the current locale via `Astro.url.pathname` rather than trusting a `lang` prop. The reason is that not every route reliably propagates `lang` through to BaseLayout — historically PageLayout passed nothing (fixed in Track A, but the workaround stays as a safety net), and special-route shells (404, etc.) also rendered without it, which caused the language switcher to compound `/en/en/en/...` on each click. The URL is the canonical source for the current locale on a static site, so any nav-aware component (switcher, locale-prefixed links, locale-keyed T9 lookups) should look at `Astro.url.pathname`. Props remain useful for *content* concerns (which post/page are we rendering) and for layout overrides; they are not the source of truth for "which locale am I in."
