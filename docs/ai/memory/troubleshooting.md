## `tools/migrate-from-hugo.mjs` in-loop body mutation invalidates match indices — Status: workaround
<!-- 2026-05-11 — discovered during image-rename pass (commit 957060e) -->
`processBody` (around line 150-183) collects all image matches via `body.matchAll(IMG_REF)` ONCE, then iterates and calls `body.replace(full, newRef)` inside the loop. Each replacement shrinks the body (`/img/wix/<long-hash>.png` → `attachments/<short>.png`), but every subsequent iteration still uses the pre-loop `m.index` to compute `body.slice(m.index + full.length)` for caption detection. After image #1 is replaced, image #2's slice starts at a stale offset that lands in the wrong text — caption-detection then fails and the file falls back to `image-N.<ext>`. Effect: roughly 66% of body-image attachments shipped with `image-N` placeholder names (80/122 in this repo's 28-post migration). **Symptom pattern**: first image per post often has a real caption-derived name; images 2+ degrade to placeholders. **Fix when migrating again**: process matches in reverse order (so positions before the just-spliced replacement remain valid), OR compute all replacements against the ORIGINAL body and apply them via index-splice (`body.slice(0, idx) + new + body.slice(idx + full.length)`) in reverse. The renamer `tools/rename-image-attachments.mjs` (added in 957060e) uses the index-splice approach as a reference. **Until fixed**: any new Hugo import will reproduce this bug — the workaround is to re-run the renamer after migration. Better: fix the migration tool first, then ingest.

## Astro 7 alpha dev server does not HMR `Header.astro` edits — Status: workaround
<!-- 2026-05-11 -->
Editing `src/components/Header.astro` while `pnpm dev` is running does **not** propagate changes to served HTML — requests keep returning the old rendered output even though the file's mtime updates and Vite reports no errors. `touch` on the file doesn't help. The only fix observed: stop the dev server and start it again. Suspected cause: the dev server runs Astro `v7.0.0-alpha.0` (logged at startup) and the component cache for layout-embedded `.astro` files appears to skip invalidation on parent-graph edits. The build (`pnpm build`) is unaffected — it always reflects the latest source. **Practical rule**: after editing any header/layout `.astro`, restart `pnpm dev` before curling/visiting to verify, otherwise you'll measure stale output and chase a non-existent bug.

## `.astro/` cache can mask remark-plugin edits on `pnpm build` — Status: workaround
<!-- 2026-05-11 -->
Edits to remark plugins (`src/utils/internallinks.ts`, `src/utils/remark-obsidian-embeds.ts`, anything wired into `astro.config.mjs#integrations`) are sometimes not picked up by `pnpm build` — the generated HTML keeps the pre-edit URLs even though the source has changed. The build reports success and no warnings. **Workaround**: `rm -rf .astro node_modules/.astro` and rebuild; the change then takes effect. Astro 7 alpha appears to cache the markdown-rendering graph independently of source mtimes. **Practical rule**: after touching any remark plugin, nuke `.astro/` before measuring the result. Same as the dev-server stale-Header issue above, but for the production build path. Do not waste time hypothesizing why the patch "didn't work" — it did, the cache is lying.

## `/projects/` and `/docs/` stubs emitted even with `optionalContentTypes.X = false` — Status: known-behaviour
<!-- 2026-05-11 -->
Setting `siteConfig.optionalContentTypes.projects = false` (and same for `docs`) disables the optional content sections but does **not** prevent astro-modular from emitting `dist/projects/index.html` and `dist/docs/index.html`. The fallback logic in `src/pages/projects/index.astro:16-29` (and the equivalent in `src/pages/projects/[...slug].astro` and the docs siblings) tries `getEntry('pages', 'projects')` to render a content-collection fallback first, and only redirects to `/404` when no such entry exists. The resulting HTML is a thin redirect stub, not a real page — visible in `dist/` and in the sitemap inputs. Harmless, but easy to mistake for a real route. **If you want clean URLs**: either provide a `src/content/pages/{de,en}/projects.md` (and `docs.md`) acting as a deliberate landing page (the fallback will then render it), or delete the `src/pages/{projects,docs}/` route trees entirely. Neither is needed right now — the redirect stubs cost ~1KB each.

## `pnpm run update` clobbers the build allowlist — Status: resolved
<!-- 2026-05-09 -->
astro-modular's `pnpm run update` (scripts/update.mjs) replaces framework files including `package.json` and `pnpm-workspace.yaml` when pulling a new theme release. After every update, re-verify the `allowBuilds:` and `onlyBuiltDependencies:` entries for `esbuild`, `sharp`, and `workerd` in `pnpm-workspace.yaml`, otherwise the next `pnpm install`/`pnpm build` aborts with `ERR_PNPM_IGNORED_BUILDS`.

**Resolved 2026-05-09**: the fork's patched `update.mjs` (commit `f3d5e52` on `MMoMM-org/astro-modular-mmomm`) added `pnpm-workspace.yaml` to USER_PATHS, so the file is now preserved across updates. The pnpm 11 allowlist note in `tools.md` is the durable reference; this entry stays as historical context for why USER_PATHS was extended.

## `pnpm create astro-modular` denied by Claude Code auto-mode classifier — Status: resolved
<!-- 2026-05-09 -->
The npm scaffolder (`create-astro-modular`) is blocked by Claude Code's auto-mode classifier as "untrusted external code execution". Use the upstream README's documented alternative: `git clone --depth=1 https://github.com/davidvkimball/astro-modular.git <scratch-dir>`, then mirror the CLI's CLEANUP step by deleting `cli/`, `.github/`, `.ref/`, and `AGENTS.md` from the clone before copying files into the target repo.

## `astro:content` virtual module unavailable in `astro.config.mjs` load graph — Status: resolved
<!-- 2026-05-09 -->
`astro.config.mjs` imports `src/utils/internallinks.ts` (for `remarkInternalLinks`). If `internallinks.ts` imports anything that pulls `astro:content` (e.g. `src/utils/i18n.ts`), config evaluation fails with `Cannot find module 'astro:content'` because the virtual module isn't initialized yet at config-load time. **Workaround**: don't import `i18n.ts` from `internallinks.ts`. Inline the few helpers needed — `postUrlFromPost(post)` at `internallinks.ts:6` mirrors `postUrl()` from `i18n.ts`. Keep this constraint visible: a future "DRY cleanup" PR that re-imports `i18n.ts` will silently break the build. The build pipeline's `generate-deployment-config.js` rewrites `astro.config.mjs` mid-build, occasionally racing config evaluation and surfacing this error spuriously on first build — second `pnpm build` succeeds.

## Language switcher multiplied `/en` prefixes (`/en/en/en/...`) — Status: resolved
<!-- 2026-05-10 -->
The first version of `pathForOtherLocale` (Header.astro) used the Header's `lang` prop to decide whether the current URL was already in the `/en/` namespace. On routes whose layouts don't propagate `lang` to BaseLayout — `PageLayout.astro`, `404.astro`, `ProjectLayout.astro`, `DocumentationLayout.astro` — the Header read the default `lang='de'` even when the URL was `/en/...`, so the helper prefixed `/en` onto an already-`/en/` URL. Each click on the switcher multiplied: `/about/` → `/en/about/` → `/en/en/about/` → `/en/en/en/about/` → ... **Fix** (commit `c306f7e`): derive the switcher's locale from `Astro.url.pathname`, not the prop, and make the strip greedy via `replace(/^(\/en)+(?=\/|$)/, '')`. **General rule**: any component that derives URL navigation must derive locale from the URL itself, never from a prop that may not be passed by upstream layouts. The `lang` prop is still useful for `<html lang>` (BaseLayout) and content (titles) but is not authoritative for routing decisions. **Follow-up**: PageLayout / ProjectLayout / DocumentationLayout / 404.astro should still pass `lang` to BaseLayout (cosmetic — `<html lang>` is wrong on those routes today), but the Header is now resilient regardless.

## Swup freezes the Header on language switch — Status: resolved
<!-- 2026-05-10 -->
Astro-modular ships `@swup/astro` with `containers: ['#swup-container']`. Anything outside that container — including the Header at `BaseLayout.astro:1119` — is **not** replaced when Swup intercepts a click. Result: clicking the language switcher swaps the page body to the new locale but leaves the Header showing the original locale's active label and switcher href, so it looks like the switch didn't work (and the second click does nothing because the href hasn't moved). **Fix**: tag any anchor that crosses a content boundary the Header depends on with `data-no-swup` so Swup's `linkSelector` excludes it and the browser does a full reload. Currently applied to the desktop + mobile language switchers in `Header.astro`. **Audit pattern**: any future header-state-dependent navigation (e.g. theme-aware links, user-preference toggles that re-render in the header) needs the same opt-out, OR the Header has to be moved inside `#swup-container` and the Swup hook in `BaseLayout.astro:1382` extended to re-bind the listeners after each transition.

## `Astro.props` destructure loses `Props` interface typing — Status: workaround
<!-- 2026-05-11 -->
In Astro 7 alpha, `const { x, y } = Astro.props` returns each field as `unknown` even when `export interface Props { x: number; y: number }` is declared above. `astro check` then floods the file with `ts(18046) 'x' is of type unknown` and `ts(2345) Argument of type 'unknown' is not assignable to ...` everywhere those fields are passed to typed helpers. **Fix**: cast at the destructure site, `const { x, y } = Astro.props as Props;`. Applied in `src/pages/posts/[page].astro:56` and `src/pages/en/posts/[page].astro:56` while migrating Phase 2 UI strings (commit `1b67aa4`) — dropped the total error count from 66 → 32. Any future `src/pages/**/*.astro` route declaring a `Props` interface should do the same. Don't bother trying `Astro.props as unknown as Props` or generic typing tricks — straight cast is what works.

## `const x: Locale = 'en'` keeps the literal type `'en'`, not `Locale` — Status: workaround
<!-- 2026-05-11 -->
In `.astro` script blocks (TS-compiled), `const lang: Locale = 'en'` narrows back to the literal `'en'` despite the annotation. Subsequent comparisons against the other branch — e.g. `lang === 'de' ? 'Zeige' : 'Showing'` — trip `ts(2367) types have no overlap`. The TS annotation is being out-narrowed by the const literal-type inference. **Fix that works**: `const lang = 'en' as Locale;` — the cast widens the value to the union type and the comparison type-checks. Applied to all 8 posts-list route files in commit `1b67aa4`. **General rule for this repo**: whenever a route hardcodes its locale at the top of the frontmatter, use the cast form, not the annotation form. Same pattern needed if/when other Locale-typed constants are declared from literals in route files.

## DE homepage was silently mixing DE+EN posts — Status: resolved (T11)
<!-- 2026-05-09 -->
`src/pages/index.astro` called `getCollection('posts')` with no `lang` filter, so `dist/index.html` rendered EN post links (`/en/posts/formatting-reference` etc.) alongside DE content. Fixed in T11 (commit `adedc6a`) by routing through `getLocalisedPosts(locale)`. **Audit pattern**: grep `getCollection('posts')` across `src/` — every call needs to be paired with a `lang` filter or explicit dual-locale awareness. Likely lurking the same way in tag pages, archive pages, or any unlocalized listing. The `[...page]` and `tag/[...tag]` route trees should be re-checked.
