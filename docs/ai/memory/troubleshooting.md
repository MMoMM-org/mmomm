## Images 404 from file-based i18n posts (`/posts/attachments/<file>.webp` instead of `/posts/<file>.webp`) ‚Äî Status: resolved
<!-- 2026-05-19 ‚Äî discovered while previewing first non-folder DE post on feat/astro-modular -->
`remarkFolderImages()` in `src/utils/internallinks.ts` only set `detectedLocale` when the post was folder-based (`*/index.md`). For a single-file post in a locale subfolder ‚Äî `src/content/posts/de/<slug>.md` referencing `![[hal9000.png]]` with images in a shared `src/content/posts/de/attachments/` ‚Äî the locale stayed `null`, so the resolver fell through to the legacy single-file branch and emitted `/posts/attachments/hal9000.webp`. But `scripts/sync-images.js:113-130` strips every `attachments/` segment for locale subfolders (and prefixes EN output with `/en`), so the file actually lands at `/posts/hal9000.webp`. Result: all wikilink images in file-based DE posts hard-404; EN posts would 404 the same way at `/en/posts/attachments/<file>.webp`. **Fix** (commit pending, this turn): added `detectFileBasedLocale(collectionIndex)` so `detectedLocale` is populated for both folder- and file-based posts; rewrote the two single-file branches to apply `langPrefix` (`/en` for EN, empty for DE) and strip the `attachments/` segment when a locale is present. Folder-based posts and legacy non-locale posts are unchanged. **Audit pattern**: any other place that constructs an asset URL from a `src/content/<collection>/...` path and assumes "no locale = no prefix, attachments stays in URL" needs the same dual-mode treatment. Suspect candidates: `src/utils/remark-obsidian-embeds.ts` (handles audio/video/PDF/SVG/.base for the same wikilinks ‚Äî uses similar shape, untested for file-based locale posts), `src/utils/images.ts` cover-image helpers, and `src/utils/feeds.ts`.

## `getSlugLangMap()` in internallinks.ts is locale-blind on duplicate slugs ‚Äî Status: known-behaviour
<!-- 2026-05-12 ‚Äî discovered while fixing cross-locale wikilink resolution -->
`buildSlugLangMap()` in `src/utils/internallinks.ts` walks `src/content/posts/`, strips the `de/`/`en/` prefix from each path, and writes `map.set(bareSlug, lang)`. For slugs that exist in both languages (e.g. `miyo-ace` lives at both `posts/de/miyo-ace/` and `posts/en/miyo-ace/`), the second `set()` call overwrites the first ‚Äî last-write-wins by `readdirSync` order, which is alphabetical on macOS so `en` typically wins. Consequence: `resolveLocaleAwarePostUrl('miyo-ace')` returns the EN URL even when the call originates from a DE post. **Symptom**: cross-post wikilinks render to `/en/posts/<slug>` from DE pages (or vice versa). **Workaround in place** (commit `4eb89eb`): the resolver trusts an explicit locale prefix in the input (`de/miyo-ace/index` ‚Üí `/posts/miyo-ace`), and the second-pass resolver in `remarkStandardLinks` skips already-resolved URLs (those without a `/` left after stripping `/posts/`). Both make the locale-blind map's wrong answers unreachable for path-prefixed wikilinks. **Real fix** (not done): change the map to key on `<lang>/<slug>` so both locales coexist, and have callers without an explicit locale fall back to the post-cache lookup or a "guess from current page" heuristic. Worth doing before any feature relies on bare-slug wikilinks (`[[miyo-ace]]`) round-tripping correctly across locales.

## `remark-obsidian-comments` doesn't strip `%%...%%` when comment contains inline code ‚Äî Status: workaround (theme-fork fix backlog'd)
<!-- 2026-05-12 ‚Äî discovered while debugging DE homepage rendering -->
`src/utils/remark-obsidian-comments.ts` calls `visit(tree, 'text', ...)` and runs `node.value.replace(/%%[\s\S]*?%%/g, '')` on each text node individually. When a comment contains inline code (`` `foo` ``), markdown parses the comment as multiple sibling AST nodes (text ‚Üí inlineCode ‚Üí text ‚Üí inlineCode ‚Üí text) and neither outer text node contains both `%%` delimiters, so the regex never matches ‚Äî the comment renders as visible plain text in the output. **Symptom**: a `%%` block at the top of a content file shows up on the rendered page in DE but works fine in EN (or vice versa) ‚Äî check whether the broken one has backticks inside the `%%`. **Workaround for content authors**: avoid backticks inside `%%...%%` comments. Use plain text for filenames / config keys / etc. (e.g. write `src/config.ts` not `` `src/config.ts` ``). **Real fix**: theme-fork enhancement to walk paragraph children, detect `%%` across sibling nodes, splice the entire range. Tracked as TaskList #55, not blocking.

## `tools/migrate-from-hugo.mjs` in-loop body mutation invalidates match indices ‚Äî Status: resolved
<!-- 2026-05-11 ‚Äî discovered during image-rename pass (957060e), fixed in 2f8fc77 -->
`processBody` collected all image matches via `body.matchAll(IMG_REF)` ONCE, then iterated and called `body.replace(full, newRef)` inside the loop. Each replacement shrank the body (`/img/wix/<long-hash>.png` ‚Üí `attachments/<short>.png`), but every subsequent iteration still used the pre-loop `m.index` to compute `body.slice(m.index + full.length)` for caption detection. After image #1 was replaced, image #2's slice started at a stale offset that landed in the wrong text ‚Äî caption-detection then failed and the file fell back to `image-N.<ext>`. Effect: roughly 66% of body-image attachments shipped with `image-N` placeholder names (80/122 in this repo's 28-post migration). Symptom pattern: first image per post often got a real caption-derived name; images 2+ degraded to placeholders.

**Resolved 2026-05-11** (commit `2f8fc77`): split processBody into two passes. Pass 1 walks matches against the ORIGINAL (immutable) body, queueing `{matchIndex, fullLen, newRef}` records. Pass 2 applies replacements in REVERSE order via string-splice (`body.slice(0, i) + new + body.slice(i + fullLen)`), so earlier indices stay valid because nothing before the splice point gets touched. Caption-detection also extended with alt-text-first and caption-BEFORE-image fallbacks. Smoke-tested on 5 known-hard posts: 32/37 images (86%) resolve to caption-derived names. Remaining 5 are content edge cases (images flanked by bullet lists, captions >80 chars) ‚Äî same residual set as the renamer. **`tools/rename-image-attachments.mjs` is now a one-time cleanup, not a permanent tax** ‚Äî future Hugo imports produce correct filenames on the first pass.

## Astro 7 alpha dev server does not HMR `Header.astro` edits ‚Äî Status: workaround
<!-- 2026-05-11 -->
Editing `src/components/Header.astro` while `pnpm dev` is running does **not** propagate changes to served HTML ‚Äî requests keep returning the old rendered output even though the file's mtime updates and Vite reports no errors. `touch` on the file doesn't help. The only fix observed: stop the dev server and start it again. Suspected cause: the dev server runs Astro `v7.0.0-alpha.0` (logged at startup) and the component cache for layout-embedded `.astro` files appears to skip invalidation on parent-graph edits. The build (`pnpm build`) is unaffected ‚Äî it always reflects the latest source. **Practical rule**: after editing any header/layout `.astro`, restart `pnpm dev` before curling/visiting to verify, otherwise you'll measure stale output and chase a non-existent bug.

## `scripts/sync-images.js` only runs at dev-server startup ‚Äî Status: known-behaviour
<!-- 2026-05-19 ‚Äî discovered while debugging EN translation pair's missing images -->
`pnpm dev` chains `node scripts/sync-images.js` once before booting Astro (see the `dev` script in `package.json`). The script walks the content tree, syncs images to `public/`, then exits. **Astro dev does not re-run it on filesystem changes** ‚Äî Astro's Vite watcher only re-bundles JS/CSS and re-syncs the content collection, not `public/`. Consequence: if you add a new image (or a new post that references new images) while `pnpm dev` is already running, the image lives in `src/content/.../attachments/` but never lands in `public/...`, so `<img>` tags 404 ‚Äî even though the wikilink resolver outputs the correct URL. **Workarounds**: re-run `node scripts/sync-images.js` manually (re-syncs in ~1‚Äì2s, no restart needed), OR restart `pnpm dev`. **Practical rule for translation pairs**: when creating an EN counterpart whose attachments folder gains new files, run the sync script before verifying ‚Äî otherwise you'll chase a wikilink-resolver bug that doesn't exist. **Real fix** (not done): wire sync-images into a Vite plugin's `configureServer` hook with a `chokidar` watch on `src/content/**/attachments/**` so additions hot-sync. Same root cause as the Astro 7 alpha new-content-file gotcha below ‚Äî both are "the dev server boots a fixed view of disk" ‚Äî but the workaround here is cheaper (manual sync, not a full server restart).

## Astro 7 alpha dev server does not pick up newly-added content files ‚Äî Status: workaround
<!-- 2026-05-12 ‚Äî discovered while testing remarkInlineTags via a new test post -->
Creating a new content collection file (e.g. `src/content/posts/<locale>/<slug>/index.md`) while `pnpm dev` is running does **not** add the new route to the dev server's routing table. Requests to the new URL return **HTTP 404** even after several seconds of HMR-trigger time, even if `astro:content` re-syncs the collection. Editing or renaming the file does not help either. The only fix observed: stop and restart `pnpm dev`. **Practical impact**: any feature-test that requires creating a new post (e.g. "does remarkInlineTags wrap my `#testtag` in a link?") needs a `pnpm build` for verification, not a dev-server hot reload ‚Äî otherwise the test URL 404s and the feature looks broken. Same Astro 7 alpha root cause as the Header.astro and `.astro/` cache issues above; the collection routing graph is built once at dev-server startup and not invalidated by collection-file creation events.

## `.astro/` cache can mask remark-plugin edits on `pnpm build` ‚Äî Status: workaround
<!-- 2026-05-11 -->
Edits to remark plugins (`src/utils/internallinks.ts`, `src/utils/remark-obsidian-embeds.ts`, anything wired into `astro.config.mjs#integrations`) are sometimes not picked up by `pnpm build` ‚Äî the generated HTML keeps the pre-edit URLs even though the source has changed. The build reports success and no warnings. **Workaround**: `rm -rf .astro node_modules/.astro` and rebuild; the change then takes effect. Astro 7 alpha appears to cache the markdown-rendering graph independently of source mtimes. **Practical rule**: after touching any remark plugin, nuke `.astro/` before measuring the result. Same as the dev-server stale-Header issue above, but for the production build path. Do not waste time hypothesizing why the patch "didn't work" ‚Äî it did, the cache is lying.

## `/projects/` and `/docs/` stubs emitted even with `optionalContentTypes.X = false` ‚Äî Status: known-behaviour
<!-- 2026-05-11 -->
Setting `siteConfig.optionalContentTypes.projects = false` (and same for `docs`) disables the optional content sections but does **not** prevent astro-modular from emitting `dist/projects/index.html` and `dist/docs/index.html`. The fallback logic in `src/pages/projects/index.astro:16-29` (and the equivalent in `src/pages/projects/[...slug].astro` and the docs siblings) tries `getEntry('pages', 'projects')` to render a content-collection fallback first, and only redirects to `/404` when no such entry exists. The resulting HTML is a thin redirect stub, not a real page ‚Äî visible in `dist/` and in the sitemap inputs. Harmless, but easy to mistake for a real route. **If you want clean URLs**: either provide a `src/content/pages/{de,en}/projects.md` (and `docs.md`) acting as a deliberate landing page (the fallback will then render it), or delete the `src/pages/{projects,docs}/` route trees entirely. Neither is needed right now ‚Äî the redirect stubs cost ~1KB each.

**Resolved 2026-05-09**: the fork's patched `update.mjs` (commit `f3d5e52` on `MMoMM-org/astro-modular-mmomm`) added `pnpm-workspace.yaml` to USER_PATHS, so the file is now preserved across updates. The pnpm 11 allowlist note in `tools.md` is the durable reference; this entry stays as historical context for why USER_PATHS was extended.

## `astro:content` virtual module unavailable in `astro.config.mjs` load graph ‚Äî Status: resolved
<!-- 2026-05-09 -->
`astro.config.mjs` imports `src/utils/internallinks.ts` (for `remarkInternalLinks`). If `internallinks.ts` imports anything that pulls `astro:content` (e.g. `src/utils/i18n.ts`), config evaluation fails with `Cannot find module 'astro:content'` because the virtual module isn't initialized yet at config-load time. **Workaround**: don't import `i18n.ts` from `internallinks.ts`. Inline the few helpers needed ‚Äî `postUrlFromPost(post)` at `internallinks.ts:6` mirrors `postUrl()` from `i18n.ts`. Keep this constraint visible: a future "DRY cleanup" PR that re-imports `i18n.ts` will silently break the build. The build pipeline's `generate-deployment-config.js` rewrites `astro.config.mjs` mid-build, occasionally racing config evaluation and surfacing this error spuriously on first build ‚Äî second `pnpm build` succeeds.

## Language switcher multiplied `/en` prefixes (`/en/en/en/...`) ‚Äî Status: resolved
<!-- 2026-05-10 -->
The first version of `pathForOtherLocale` (Header.astro) used the Header's `lang` prop to decide whether the current URL was already in the `/en/` namespace. On routes whose layouts don't propagate `lang` to BaseLayout ‚Äî `PageLayout.astro`, `404.astro`, `ProjectLayout.astro`, `DocumentationLayout.astro` ‚Äî the Header read the default `lang='de'` even when the URL was `/en/...`, so the helper prefixed `/en` onto an already-`/en/` URL. Each click on the switcher multiplied: `/about/` ‚Üí `/en/about/` ‚Üí `/en/en/about/` ‚Üí `/en/en/en/about/` ‚Üí ... **Fix** (commit `c306f7e`): derive the switcher's locale from `Astro.url.pathname`, not the prop, and make the strip greedy via `replace(/^(\/en)+(?=\/|$)/, '')`. **General rule**: any component that derives URL navigation must derive locale from the URL itself, never from a prop that may not be passed by upstream layouts. The `lang` prop is still useful for `<html lang>` (BaseLayout) and content (titles) but is not authoritative for routing decisions. **Follow-up**: PageLayout / ProjectLayout / DocumentationLayout / 404.astro should still pass `lang` to BaseLayout (cosmetic ‚Äî `<html lang>` is wrong on those routes today), but the Header is now resilient regardless.

## Language switcher href stays stale across Swup intra-locale navigation ‚Äî Status: resolved
<!-- 2026-05-20 ‚Äî user reported "clicking EN on a post lands on /en/ but only when I arrived via click from /, not via direct URL" -->
Same root-cause family as the "Swup freezes Header on language switch" entry below, but the symptom and fix are different. **Setup**: Astro-modular ships Swup with `containers: ['#swup-container']`. The `<header id="main-header">` element sits OUTSIDE that container, so its server-rendered DOM (including the `<a hreflang="en">` language-switcher's `href`) is **never replaced** during a Swup AJAX transition. **Symptom**: land on `/` (DE home) ‚Üí switcher renders with `href="/en/"` (the home's correct counterpart) ‚Üí click an internal post link ‚Üí Swup navigates ‚Üí URL becomes the post URL, body updates, BUT the header's switcher still points at `/en/`. Hovering shows `/en/`; clicking lands on the EN homepage instead of the post's EN counterpart. Direct URL entry shows the correct switcher because the Header is re-rendered server-side. **Diagnostic giveaway**: behaviour depends on **how** the user arrived, not on which page they're on. **Fix** (commit `13ab950`): in the Swup `visit:end` hook, re-read `<link rel="alternate" hreflang="...">` from the freshly-updated `<head>` (Swup's `updateHead: true` already replaces head tags) and write each `#main-header a[hreflang]` switcher's `href` to match. Pages without a translation pair emit no hreflang link ‚Üí fall back to the locale root (matches the server-side fallback path in `Header.astro`). **General principle**: any Header-rendered state that depends on the current page's context ‚Äî switcher href, page-specific labels, "current page" highlighting derived from URL ‚Äî needs an explicit re-sync in a Swup hook because the Header DOM is preserved verbatim across transitions. Two instances of this class so far; treat as a pattern. **Alternatives that were rejected**: (a) move Header into `#swup-container` ‚Äî needs re-binding mobile menu, theme button, dropdown handlers after each transition; (b) `data-no-swup` on every internal link ‚Äî defeats Swup. The visit:end-hook approach is the lightest touch.

## Swup freezes the Header on language switch ‚Äî Status: resolved
<!-- 2026-05-10 -->
Astro-modular ships `@swup/astro` with `containers: ['#swup-container']`. Anything outside that container ‚Äî including the Header at `BaseLayout.astro:1119` ‚Äî is **not** replaced when Swup intercepts a click. Result: clicking the language switcher swaps the page body to the new locale but leaves the Header showing the original locale's active label and switcher href, so it looks like the switch didn't work (and the second click does nothing because the href hasn't moved). **Fix**: tag any anchor that crosses a content boundary the Header depends on with `data-no-swup` so Swup's `linkSelector` excludes it and the browser does a full reload. Currently applied to the desktop + mobile language switchers in `Header.astro`. **Audit pattern**: any future header-state-dependent navigation (e.g. theme-aware links, user-preference toggles that re-render in the header) needs the same opt-out, OR the Header has to be moved inside `#swup-container` and the Swup hook in `BaseLayout.astro:1382` extended to re-bind the listeners after each transition.

## `Astro.props` destructure loses `Props` interface typing ‚Äî Status: workaround
<!-- 2026-05-11 -->
In Astro 7 alpha, `const { x, y } = Astro.props` returns each field as `unknown` even when `export interface Props { x: number; y: number }` is declared above. `astro check` then floods the file with `ts(18046) 'x' is of type unknown` and `ts(2345) Argument of type 'unknown' is not assignable to ...` everywhere those fields are passed to typed helpers. **Fix**: cast at the destructure site, `const { x, y } = Astro.props as Props;`. Applied in `src/pages/posts/[page].astro:56` and `src/pages/en/posts/[page].astro:56` while migrating Phase 2 UI strings (commit `1b67aa4`) ‚Äî dropped the total error count from 66 ‚Üí 32. Any future `src/pages/**/*.astro` route declaring a `Props` interface should do the same. Don't bother trying `Astro.props as unknown as Props` or generic typing tricks ‚Äî straight cast is what works.

## `const x: Locale = 'en'` keeps the literal type `'en'`, not `Locale` ‚Äî Status: workaround
<!-- 2026-05-11 -->
In `.astro` script blocks (TS-compiled), `const lang: Locale = 'en'` narrows back to the literal `'en'` despite the annotation. Subsequent comparisons against the other branch ‚Äî e.g. `lang === 'de' ? 'Zeige' : 'Showing'` ‚Äî trip `ts(2367) types have no overlap`. The TS annotation is being out-narrowed by the const literal-type inference. **Fix that works**: `const lang = 'en' as Locale;` ‚Äî the cast widens the value to the union type and the comparison type-checks. Applied to all 8 posts-list route files in commit `1b67aa4`. **General rule for this repo**: whenever a route hardcodes its locale at the top of the frontmatter, use the cast form, not the annotation form. Same pattern needed if/when other Locale-typed constants are declared from literals in route files.

## DE homepage was silently mixing DE+EN posts ‚Äî Status: resolved (T11)
<!-- 2026-05-09 -->
`src/pages/index.astro` called `getCollection('posts')` with no `lang` filter, so `dist/index.html` rendered EN post links (`/en/posts/formatting-reference` etc.) alongside DE content. Fixed in T11 (commit `adedc6a`) by routing through `getLocalisedPosts(locale)`. **Audit pattern**: grep `getCollection('posts')` across `src/` ‚Äî every call needs to be paired with a `lang` filter or explicit dual-locale awareness. Likely lurking the same way in tag pages, archive pages, or any unlocalized listing. The `[...page]` and `tag/[...tag]` route trees should be re-checked.

**Audit instances fixed 2026-05-19** (confirms the T11 prediction; same root cause, same class of bug, three more surfaces):
- `src/layouts/PostLayout.astro` ‚Äî older/newer post navigation crossed locales because `visiblePosts` was passed to `getAdjacentPosts` without a `lang` filter (commit `6462b1d`). Filter to `post.data.lang === currentPost.data.lang` before computing adjacency.
- `src/components/LinkedMentions.astro` ‚Äî backlinks listed mentions from every locale; PostLayout already threads `lang` as a prop, just needed to filter `posts` and `pages` by it before passing into `findLinkedMentions` (commit `ae9e909`).
- `src/pages/api/posts.json.ts` + `pages.json.ts` + `src/components/CommandPalette.astro` ‚Äî the static JSON endpoints are built once at the root URL so they legitimately must include every locale; the right fix is to expose `lang` on each item and have the client-side palette filter by `window.location.pathname.startsWith('/en/')`. The Header switcher already uses pathname (not a prop) for the same reason ‚Äî commit `c306f7e` codified that. Items without `lang` (projects, docs) pass through (commit `ae9e909`).

**Audit instances reviewed-and-skipped**:
- `src/layouts/ProjectLayout.astro` ‚Äî same locale-blind shape as PostLayout (`getCollection('posts')` for wikilink resolution), but projects are disabled site-wide via `siteConfig.optionalContentTypes.projects = false`, so no current user impact. Re-test if projects are ever enabled.
- `src/pages/sitemap.xml.ts` ‚Äî legitimately needs both locales for `<xhtml:link rel="alternate" hreflang>` fan-out; this is the only reasonable use of cross-locale `getCollection('posts')` in this codebase.
- `src/components/LocalGraph.astro` ‚Äî the frontmatter's `getCollection('posts')` was dead code; the actual graph data is loaded client-side from `/graph/graph-data.json`, which is generated by `scripts/generate-graph-data.js` and tags every node with `lang`. Frontmatter dead code removed (commit `ae9e909`); graph generator already locale-aware.

**Heuristic for future PRs**: any new `getCollection('posts')` call in `src/` should be one of three forms ‚Äî `getLocalisedPosts(locale)`, a route-derived `lang` filter, or an explicit `// cross-locale by design ‚Äî <why>` comment. Same shape applies to `getCollection('pages')`.
