## `/projects/` and `/docs/` stubs emitted even with `optionalContentTypes.X = false` ÔÇö Status: known-behaviour
<!-- 2026-05-11 -->
Setting `siteConfig.optionalContentTypes.projects = false` (and same for `docs`) disables the optional content sections but does **not** prevent astro-modular from emitting `dist/projects/index.html` and `dist/docs/index.html`. The fallback logic in `src/pages/projects/index.astro:16-29` (and the equivalent in `src/pages/projects/[...slug].astro` and the docs siblings) tries `getEntry('pages', 'projects')` to render a content-collection fallback first, and only redirects to `/404` when no such entry exists. The resulting HTML is a thin redirect stub, not a real page ÔÇö visible in `dist/` and in the sitemap inputs. Harmless, but easy to mistake for a real route. **If you want clean URLs**: either provide a `src/content/pages/{de,en}/projects.md` (and `docs.md`) acting as a deliberate landing page (the fallback will then render it), or delete the `src/pages/{projects,docs}/` route trees entirely. Neither is needed right now ÔÇö the redirect stubs cost ~1KB each.

## `pnpm run update` clobbers the build allowlist ÔÇö Status: resolved
<!-- 2026-05-09 -->
astro-modular's `pnpm run update` (scripts/update.mjs) replaces framework files including `package.json` and `pnpm-workspace.yaml` when pulling a new theme release. After every update, re-verify the `allowBuilds:` and `onlyBuiltDependencies:` entries for `esbuild`, `sharp`, and `workerd` in `pnpm-workspace.yaml`, otherwise the next `pnpm install`/`pnpm build` aborts with `ERR_PNPM_IGNORED_BUILDS`.

**Resolved 2026-05-09**: the fork's patched `update.mjs` (commit `f3d5e52` on `MMoMM-org/astro-modular-mmomm`) added `pnpm-workspace.yaml` to USER_PATHS, so the file is now preserved across updates. The pnpm 11 allowlist note in `tools.md` is the durable reference; this entry stays as historical context for why USER_PATHS was extended.

## `pnpm create astro-modular` denied by Claude Code auto-mode classifier ÔÇö Status: resolved
<!-- 2026-05-09 -->
The npm scaffolder (`create-astro-modular`) is blocked by Claude Code's auto-mode classifier as "untrusted external code execution". Use the upstream README's documented alternative: `git clone --depth=1 https://github.com/davidvkimball/astro-modular.git <scratch-dir>`, then mirror the CLI's CLEANUP step by deleting `cli/`, `.github/`, `.ref/`, and `AGENTS.md` from the clone before copying files into the target repo.

## `astro:content` virtual module unavailable in `astro.config.mjs` load graph ÔÇö Status: resolved
<!-- 2026-05-09 -->
`astro.config.mjs` imports `src/utils/internallinks.ts` (for `remarkInternalLinks`). If `internallinks.ts` imports anything that pulls `astro:content` (e.g. `src/utils/i18n.ts`), config evaluation fails with `Cannot find module 'astro:content'` because the virtual module isn't initialized yet at config-load time. **Workaround**: don't import `i18n.ts` from `internallinks.ts`. Inline the few helpers needed ÔÇö `postUrlFromPost(post)` at `internallinks.ts:6` mirrors `postUrl()` from `i18n.ts`. Keep this constraint visible: a future "DRY cleanup" PR that re-imports `i18n.ts` will silently break the build. The build pipeline's `generate-deployment-config.js` rewrites `astro.config.mjs` mid-build, occasionally racing config evaluation and surfacing this error spuriously on first build ÔÇö second `pnpm build` succeeds.

## Language switcher multiplied `/en` prefixes (`/en/en/en/...`) ÔÇö Status: resolved
<!-- 2026-05-10 -->
The first version of `pathForOtherLocale` (Header.astro) used the Header's `lang` prop to decide whether the current URL was already in the `/en/` namespace. On routes whose layouts don't propagate `lang` to BaseLayout ÔÇö `PageLayout.astro`, `404.astro`, `ProjectLayout.astro`, `DocumentationLayout.astro` ÔÇö the Header read the default `lang='de'` even when the URL was `/en/...`, so the helper prefixed `/en` onto an already-`/en/` URL. Each click on the switcher multiplied: `/about/` Ôćĺ `/en/about/` Ôćĺ `/en/en/about/` Ôćĺ `/en/en/en/about/` Ôćĺ ... **Fix** (commit `c306f7e`): derive the switcher's locale from `Astro.url.pathname`, not the prop, and make the strip greedy via `replace(/^(\/en)+(?=\/|$)/, '')`. **General rule**: any component that derives URL navigation must derive locale from the URL itself, never from a prop that may not be passed by upstream layouts. The `lang` prop is still useful for `<html lang>` (BaseLayout) and content (titles) but is not authoritative for routing decisions. **Follow-up**: PageLayout / ProjectLayout / DocumentationLayout / 404.astro should still pass `lang` to BaseLayout (cosmetic ÔÇö `<html lang>` is wrong on those routes today), but the Header is now resilient regardless.

## Swup freezes the Header on language switch ÔÇö Status: resolved
<!-- 2026-05-10 -->
Astro-modular ships `@swup/astro` with `containers: ['#swup-container']`. Anything outside that container ÔÇö including the Header at `BaseLayout.astro:1119` ÔÇö is **not** replaced when Swup intercepts a click. Result: clicking the language switcher swaps the page body to the new locale but leaves the Header showing the original locale's active label and switcher href, so it looks like the switch didn't work (and the second click does nothing because the href hasn't moved). **Fix**: tag any anchor that crosses a content boundary the Header depends on with `data-no-swup` so Swup's `linkSelector` excludes it and the browser does a full reload. Currently applied to the desktop + mobile language switchers in `Header.astro`. **Audit pattern**: any future header-state-dependent navigation (e.g. theme-aware links, user-preference toggles that re-render in the header) needs the same opt-out, OR the Header has to be moved inside `#swup-container` and the Swup hook in `BaseLayout.astro:1382` extended to re-bind the listeners after each transition.

## DE homepage was silently mixing DE+EN posts ÔÇö Status: resolved (T11)
<!-- 2026-05-09 -->
`src/pages/index.astro` called `getCollection('posts')` with no `lang` filter, so `dist/index.html` rendered EN post links (`/en/posts/formatting-reference` etc.) alongside DE content. Fixed in T11 (commit `adedc6a`) by routing through `getLocalisedPosts(locale)`. **Audit pattern**: grep `getCollection('posts')` across `src/` ÔÇö every call needs to be paired with a `lang` filter or explicit dual-locale awareness. Likely lurking the same way in tag pages, archive pages, or any unlocalized listing. The `[...page]` and `tag/[...tag]` route trees should be re-checked.
